# VDA 5050 v2.1.0 — Overview & Terminology

Source: *Interface for the communication between automated guided vehicles
(AGV) and a master control*, VDA 5050, Version 2.1.0, January 2025.
Published jointly by VDA (Verband der Automobilindustrie) and VDMA. Spec
repo: https://github.com/vda5050/vda5050. Local copy:
`VDA5050-V2.1.0-2025-01-1-4.pdf` (this skill folder).

---

## 1. Purpose

VDA 5050 standardizes the communication interface between a **master
control** (fleet manager / coordination system) and **AGVs** within a
**driverless transport system (DTS)**, so that AGVs from different
manufacturers can be mixed under one master control. It covers:

- Sending **orders** (driving routes + actions) from master control to AGV.
- Sending **instant actions** (immediate, out-of-band commands).
- Reporting AGV **state** (position, progress, battery, errors, action
  status) back to master control.
- Exchanging **factsheet** (AGV capabilities/limits) and **connection**
  (liveness) information.
- Distributing and managing **maps**.

Goals: plug-and-play vehicle integration, manufacturer independence,
support for ≥1000 vehicles, mixed autonomy levels, and traffic/intersection
decision-making by master control.

Transport: **MQTT** (3.1.1 minimum) + **JSON** payloads. Out of scope:
AGV-to-peripheral communication (e.g., AGV↔gate), route network
configuration/CAD import, and full freely-navigating-vehicle path planning
(only the order interface is specified).

Related documents: VDI 2510 (DTS), VDI 4451 Sheet 7 (DTS master control
compatibility), DIN EN ISO 3691-4 (driverless truck safety), **LIF — Layout
Interchange Format** (track layout exchange between integrator and master
control, March 2024).

## 2. Core terminology (glossary)

| Term | Meaning |
|---|---|
| **AGV** | Automated Guided Vehicle. In this spec's narrower usage, a vehicle whose path planning is *not* fully autonomous — it follows guidance or master-control-provided routes (contrast with "freely navigating"/autonomous vehicles). |
| **DTS** | Driverless Transport System — the overall system of master control + AGVs + (optionally) a proprietary master control bridging legacy stock systems. |
| **Master control** | The coordination-plane system that assigns orders, calculates/guides routes, manages traffic/deadlocks/energy, and talks to peripherals (doors, elevators). |
| **Free navigation AGV** | Vehicle that plans its own path from start/destination coordinates; reports its path to master control; can keep moving if master control connection breaks; may bypass obstacles itself. |
| **Guided vehicle** (physical or virtual) | Vehicle whose path is calculated by master control; on connection loss it finishes its released **base** then stops; may still locally fine-adjust or bypass obstacles. |
| **Central map** | The map held centrally by master control (created once, then used/distributed). |
| **Order** | A graph of `nodes` + `edges` (with attached `actions`) sent by master control telling the AGV what to traverse/do. |
| **Base** | The released (binding) portion of an order's nodes/edges — the AGV is expected to have already executed it or is currently executing it; master control assumes it cannot be changed. |
| **Horizon** | The unreleased (`released: false`) portion of an order — route master control currently *plans* but hasn't committed to; can still be changed/replaced/deleted. |
| **Decision point** | The last node of the base; the AGV stops here if no further base is supplied before it arrives. |
| **orderId** | Identifies a (possibly multi-message) order; all messages of one order share this. |
| **orderUpdateId** | uint32, unique per `orderId`, incremented with each update/extension to the same order; used to detect stale/duplicate/deprecated messages. |
| **sequenceId** | uint32 running across *all* nodes+edges of one order (not reset on update), used to disambiguate a node/edge visited more than once (loops) and to track order progress; assigned once and never changes across updates. |
| **Stitching** | The mechanism where an order update's first (base) node must equal the previous message's last base node, so the AGV can "stitch" the update onto what it already knows without re-sending unchanged nodes. |
| **released** (node/edge) | Boolean. `true` = part of the base (AGV must traverse). `false` = part of the horizon (AGV must NOT traverse yet). An edge can only be released if both its endpoints are released; no released items may follow an unreleased one in sequence. |
| **node** | A graph vertex: `nodeId`, `sequenceId`, optional `nodeDescription`, `released`, optional `nodePosition`, `actions[]`. |
| **edge** | A directed connection between two nodes: `edgeId`, `sequenceId`, `startNodeId`/`endNodeId`, optional `maxSpeed`/`maxHeight`/`minHeight`/`orientation`/`orientationType`/`direction`/`rotationAllowed`/`maxRotationSpeed`/`trajectory`/`corridor`, `actions[]`. |
| **corridor** | Optional edge attribute defining left/right deviation boundaries (in meters) within which the AGV may leave its predefined trajectory to avoid obstacles, referenced to either the vehicle's kinematic center or its contour (`corridorRefPoint`). |
| **trajectory** | Optional NURBS curve (degree, knotVector, controlPoints) describing the actual path an AGV should/will follow between two nodes. |
| **allowedDeviationXY / allowedDeviationTheta** | Per-node tolerance radius (m) / orientation tolerance (rad) within which the AGV's position/heading still counts as "at the node" / "traversed". |
| **zoneSetId** | Optional identifier of the zone set the AGV must use for navigation/planning (omit entirely if zones aren't used). |
| **action** | A discrete behavior the AGV performs, attached to a node, an edge, or sent standalone as an instant action. Has `actionType`, `actionId` (unique, suggest UUID), optional `actionDescription`, `blockingType`, optional `actionParameters[]`. |
| **blockingType** | Enum `NONE` (can run in parallel with driving/other actions), `SOFT` (parallel with other actions, but vehicle must not drive), `HARD` (no other action may run in parallel, vehicle must not drive). |
| **actionStatus** | Lifecycle state of an action: `WAITING` → `INITIALIZING` → `RUNNING` → (`PAUSED` ↔ `RUNNING`) → `FINISHED` / `FAILED`. See state machine in §8. |
| **instantAction** | An action sent via the separate `instantActions` topic to be executed immediately, independent of (and non-conflicting with) the current order. |
| **map** | A floor/area-specific coordinate space, identified by `mapId` (+ `mapVersion`); all maps share the same project-specific global origin. |
| **mapStatus** | `ENABLED` (in active use; only one version per `mapId` may be enabled) or `DISABLED`. |
| **headerId** | uint32, per-topic counter incremented by 1 on every *sent* message (not necessarily every received one) — part of every message's header. |
| **operatingMode** | AGV's control mode: `AUTOMATIC`, `SEMIAUTOMATIC`, `MANUAL`, `SERVICE`, `TEACHIN` (see §9.1 table). |
| **eStop** | Emergency-stop acknowledge type in `safetyState`: `AUTOACK`, `MANUAL`, `REMOTE`, `NONE`. |
| **connectionState** | `ONLINE`, `OFFLINE` (graceful disconnect), `CONNECTIONBROKEN` (unexpected loss, via MQTT last-will). |
| **newBaseRequest** | Boolean flag the AGV sets `true` when its base is running low, signaling master control to extend it. |
| **factsheet** | AGV-published, mostly-static document describing the vehicle's type, physical limits, protocol limits/features, geometry, load handling, and software/network config — used by master control for planning/integration. |
| **NURBS** | Non-Uniform Rational B-Spline — the curve representation used for `trajectory` (edges) and envelope curves. |

## 3. Architecture & message flow

```
 driverless transport system (DTS)
 ┌───────────────────────────────────────────────────────────┐
 │  Master control for AGVs  ───(VDA5050 / MQTT)───  AGVs      │
 │       ↕ (supplier-specific, optional)                       │
 │  proprietary master control (legacy system bridge)          │
 └───────────────────────────────────────────────────────────┘
```

Operational sequence (steady state):

```
Master Control          MQTT Broker            AGV
      |<----------------- AGV state ------------|
 [Order Control]
 [Driving course assignment]
      |---- order (nodes+edges+actions) -------->|
      |<----------------- AGV state ------------|
      |---- instant action --------------------->|
      |<----------------- AGV state -- (carrying out) |
```

During **implementation/configuration** (not runtime): routes are CAD-imported
or manually entered into master control; route network (stations, charging
spots, peripherals, waiting/buffer positions) is configured; AGV physical
properties are obtained from the AGV's `factsheet`. Map coordinate systems
between master control and AGV must be matched, and a consistent pivot
(reference) point chosen (it may differ loaded vs. unloaded).

Master control responsibilities: assign orders; calculate/guide routes
within each AGV's physical limits; detect/resolve deadlocks; manage energy
(charging can interrupt transfer orders); control traffic (buffer routes,
waiting positions); handle temporary environment changes; talk to
peripherals (doors/gates/elevators); detect/resolve comm errors.

AGV responsibilities: localize itself; navigate its assigned route (guided
or autonomous); execute actions; continuously report state.

## 4. MQTT transport details

- **Protocol**: MQTT 3.1.1 minimum. JSON payload, UTF-8.
- **QoS**: 0 (best effort) for `order`, `instantActions`, `state`,
  `factsheet`, `visualization`. **QoS 1** (at least once) for `connection`.
- **Topic structure** (locally suggested; mandatory part is the topic name
  itself — cloud brokers may need a different prefix):
  `interfaceName/majorVersion/manufacturer/serialNumber/topic`
  e.g. `uagv/v2/KIT/0001/order`.
  - `interfaceName`: name of the interface used.
  - `majorVersion`: VDA5050 major version, prefixed with `v` (e.g. `v2`).
  - `manufacturer`, `serialNumber`: identify the AGV. Allowed chars for IDs
    generally: `A-Z a-z 0-9 _ - . :` — never `/` (topic separator) or `$`
    (reserved by some brokers).
- **Versioning**: semantic `[Major].[Minor].[Patch]` in every message's
  `version` field. Major = breaking change (new non-optional field), Minor =
  new feature (e.g. new topic), Patch = precision/behavior tweak.
- **Message length**: not capped by the spec; bounded by MQTT and by the
  AGV's `factsheet.protocolLimits`. If memory is insufficient the AGV must
  reject the order.
- **Formatting conventions**: all field names camelCase; all enum values
  UPPERCASE, no underscores; booleans are JSON `true`/`false` (never
  `'TRUE'`); no NaN/Infinity.
- **Optional fields**: master control must only send optional fields the AGV
  actually supports (per its factsheet); if an AGV receives a field it can't
  handle, it must reject the order and report an error — it cannot silently
  ignore it.

Every JSON message begins with this (non-JSON-object, conceptually a)
**header**:

| Field | Type | Description |
|---|---|---|
| `headerId` | uint32 | Per-topic counter, +1 each *sent* message. |
| `timestamp` | string | ISO 8601 UTC, e.g. `2017-04-15T11:40:03.12Z`. |
| `version` | string | Protocol version `Major.Minor.Patch`. |
| `manufacturer` | string | AGV manufacturer. |
| `serialNumber` | string | AGV serial number. |

## 6. Topic: `order` (master control → AGV)

### 6.1 Concept

An order is a graph of nodes/edges that the AGV must traverse in list order.
Master control holds the *full* graph (with per-AGV traversal restrictions);
the order message sent to a specific AGV only contains the nodes/edges that
AGV is allowed to use — restrictions themselves are never communicated.

Rules for a valid order:
- ≥1 node; `#edges == #nodes - 1`.
- The first node must be trivially reachable (AGV already standing on it, or
  within its deviation range).
- An edge can only be `released` if both its `startNodeId` and `endNodeId`
  nodes are released.
- No released node/edge may follow an unreleased one in sequence.
- It's valid to send an order with no horizon at all.
- A full transport order can be split into multiple order messages
  ("sub-orders") connected by shared `orderId`/incrementing `orderUpdateId`.

### 6.2 Orders and order updates

- **Base**: the released route the AGV is committed to. Its last node is the
  **decision point** — the AGV stops there if the base isn't extended in time.
- **Horizon**: master control's currently-planned (but unreleased, hence
  changeable/replaceable/deletable) continuation.
- MQTT is asynchronous/unreliable, so the **base cannot be retracted** —
  master control must assume it's already been executed.
- To extend/modify the horizon, master control sends an order update: same
  `orderId`, incremented `orderUpdateId`, where the **first node of the
  update equals the last base node of the previous message** (stitching).
  Only that one stitching node is resent; its contents (actions, deviation
  range, etc.) must not be changed from the original — the AGV uses what was
  first specified for it.
- `sequenceId` runs across all nodes+edges of one order and is **not reset**
  by an update — it lets the AGV distinguish a node revisited via a loop and
  lets master control unambiguously refer to a specific traversal.

Example pseudocode (order then order update):
```
// t=1: initial order
{ orderId: "1234", orderUpdateId: 0,
  nodes: [ f{released:true}, d{released:true}, g{released:true},
           b{released:false}, h{released:false} ],
  edges: [ e1{released:true}, e3{released:true},
           e8{released:false}, e9{released:false} ] }

// t=3: order update — extends horizon, stitches on node g
{ orderId: "1234", orderUpdateId: 1,
  nodes: [ g{released:true}, b{released:true}, h{released:true}, i{released:false} ],
  edges: [ e8{released:true}, e9{released:true}, e10{released:false} ] }
```

### 6.3 Order acceptance / rejection logic

On receipt, the AGV evaluates (see Figure 8 decision tree in the spec):

1. **Is the JSON valid?** No → reject, throw `validationError`.
2. **New order or update?** (`orderId` differs from current → new order.)
3. *(New order path)* **Still executing/awaiting update?** → reject (vehicle
   busy). **Is start node reachable?** → reject if not. Otherwise: delete
   previous order's states, accept (set `orderId`/`orderUpdateId`, populate
   states), execute.
4. *(Update path)* **Is `orderUpdateId` deprecated** (smaller than current)?
   → reject, `orderUpdateError`, keep executing the previous order.
   **Already on vehicle** (same `orderUpdateId` received twice)? → silently
   discard (likely a master-control resend due to a late state ack).
   Otherwise check: **valid continuation of the running order** (first node
   == current decision point) or **valid continuation of a just-completed
   order** (first `nodeId`+`sequenceId` == AGV's `lastNodeId`+
   `lastNodeSequenceId`)? If neither → reject. If yes → accept, clear horizon
   if applicable, append/populate states, execute.

Rejection scenarios and resolutions:
- **Malformed order**: don't buffer it; report warning `validationError`
  until a valid order is accepted.
- **Order with unsupported actions/fields** (e.g. lift height too high,
  trajectory the AGV can't use): don't buffer it; report warning
  `orderError` referencing the offending fields, until accepted.
- **Same `orderId`, lower `orderUpdateId`**: don't buffer it; keep the
  previous order; report `orderUpdateError`; keep executing the previous
  order.
- **Identical `orderId`+`orderUpdateId` resent**: second copy is ignored
  (handles master control resending after a late/lost state ack).

### 6.4 Order cancellation (`cancelOrder` instantAction)

On `cancelOrder`: waiting actions are immediately set `FAILED`; running
actions are interrupted and set `FAILED` if interruptible, otherwise allowed
to finish (`RUNNING` → `FINISHED`/`FAILED`); the AGV either stops immediately
or drives on to the next node (if it can't stop mid-edge) — `cancelOrder`'s
own actionState is `RUNNING` until the vehicle and all its actions have
fully stopped, then `FINISHED`. `lastNodeId`/`lastNodeSequenceId` are updated
on arrival. All `nodeStates`/`edgeStates` are deleted; `actionStates` are
kept. `orderId`/`orderUpdateId` are retained.

To receive a new order after cancellation: either (a) send an order whose
first node is a temporary node positioned where the AGV currently stands
(trivially reachable), or (b) send an order whose first node is the AGV's
last traversed node with a large enough deviation range to count the AGV as
already there. Both must be supported by the AGV.

If `cancelOrder` arrives but the AGV has no order (or the order was already
canceled): report the action `FAILED`, plus a `noOrderToCancel` `WARNING`
error referencing the action's `actionId` as `errorReference`.

### 6.5 Corridors

The optional `corridor` edge attribute lets a normally line-guided-style
vehicle deviate from its predefined/sent trajectory (within `leftWidth` /
`rightWidth` meters) to avoid obstacles, referenced to either
`KINEMATICCENTER` or `CONTOUR` via `corridorRefPoint`. The vehicle must
continuously verify it's within bounds; if not, it stops and reports an
error — master control then decides whether to require user interaction or
send a new order with adjusted corridor info. Master control should omit
`corridor` entirely (not set it to zero) if no deviation is allowed, and
should account for the larger swept footprint when making traffic decisions.

### 6.6 `order` message structure

Top-level (after the header): `orderId` (string), `orderUpdateId` (uint32),
*optional* `zoneSetId` (string — omit if unused), `nodes[node]` (array, ≥1),
`edges[edge]` (array, possibly empty).

**node**: `nodeId` (string), `sequenceId` (uint32), *optional*
`nodeDescription`, `released` (bool), *optional* `nodePosition`,
`actions[action]` (array, possibly empty).

**nodePosition**: `x`, `y` (m, float64), *optional* `theta` (rad,
[-Pi..Pi] — if set, AGV must assume this heading at the node; rotates on the
node if the previous edge disallows rotation, or if the following edge wants
a different orientation but also disallows rotation on-edge), *optional*
`allowedDeviationXY` (m; 0 = no deviation beyond normal tolerance), optional
`allowedDeviationTheta` (rad, [0..Pi]), `mapId` (string — each map shares the
project's global coordinate origin; AGV "jumps" maps e.g. on elevators),
optional `mapDescription`.

**edge**: `edgeId` (string), `sequenceId` (uint32), optional
`edgeDescription`, `released` (bool), `startNodeId`, `endNodeId` (string),
optional `maxSpeed` (m/s), `maxHeight`/`minHeight` (m), `orientation` (rad —
meaning depends on `orientationType`), `orientationType` (`GLOBAL` |
`TANGENTIAL`, default `TANGENTIAL`; 0 = forward, Pi = backward when
tangential), `direction` (free string hint for junctions, e.g.
"left"/"right"/"straight"), `rotationAllowed` (bool; if `false` and the AGV
isn't already correctly oriented, it must rotate before entering the edge or
reject the order), `maxRotationSpeed` (rad/s), `trajectory` (NURBS object,
optional — omit if AGV can't process or plans its own), `length` (m, used by
line-guided AGVs to brake in time), `corridor` (object), `action[action]`
(array — active only while the AGV is on that edge; state reverts when the
edge is left).

**trajectory**: `degree` (float64, default 1), `knotVector[float64]` (range
0.0–1.0, length = #controlPoints + degree + 1), `controlPoints[controlPoint]`
(explicitly includes start/end points).

**controlPoint**: `x`, `y` (float64, world coords), optional `weight`
(default 1.0).

**corridor**: `leftWidth`, `rightWidth` (m, float64 ≥0), optional
`corridorRefPoint` (`KINEMATICCENTER` | `CONTOUR`, default
`KINEMATICCENTER`).

**action**: `actionType` (string, matches a predefined action name or a
manufacturer-defined one), `actionId` (string, unique — suggest UUID),
optional `actionDescription`, `blockingType` (`NONE`|`SOFT`|`HARD`),
optional `actionParameters[actionParameter]`.

**actionParameter**: `{key: string, value: array|boolean|number|string|object}`.

## 7. Maps

Coordinate system: right-handed, **z-axis up**, positive rotation =
counterclockwise; vehicle frame also right-handed with **x forward**,
vehicle reference point = (0,0,0) unless stated otherwise (per DIN ISO 8855
§2.11). X/Y in meters, orientation in radians within [-Pi, Pi]. All maps for
a site/location share one global origin; `mapId` differentiates floors/areas.

### 7.1 Map distribution

Maps live on a dedicated **map server**; transfer to the AGV is a **pull**,
triggered by master control via the `downloadMap` instantAction (parameters
`mapId`, `mapVersion`, `mapDownloadLink`, optional `mapHash`). Each map =
`mapId` + `mapVersion`. Sequence: trigger `downloadMap` → AGV requests/pulls
the file from the map server → AGV verifies + processes it → reports the
instant action `FINISHED` → AGV sends updated `state` with the new entry in
`maps[]`. The vehicle must reject downloading a `mapId`+`mapVersion`
combination it already has (report `FAILED`); master control must delete
first, then re-trigger download, to replace a map.

### 7.2 Enable / delete maps

- **Enable**: `enableMap` instantAction (`mapId`, `mapVersion`) sets that
  version `ENABLED` and any other version sharing the same `mapId`
  `DISABLED`. Maps can also be enabled manually on the vehicle directly — the
  result is still reported via `state`. At most one version per `mapId` may
  be `ENABLED` at a time; different `mapId`s can be enabled simultaneously.
  It's master control's job to ensure the right maps are active before
  sending orders that reference them (via `nodePosition.mapId`). To force a
  position on a newly-activated map, use `initPosition`.
- **Delete**: `deleteMap` instantAction (`mapId`, `mapVersion`) removes a map
  from vehicle memory — only master control may trigger deletion (the
  vehicle isn't allowed to self-delete, though it can report low memory to
  prompt master control to do so). After deletion, remove that entry from
  the vehicle's `state.maps[]`.
- `state.maps[]` entries: `{mapId, mapVersion, mapDescription?, mapStatus:
  'ENABLED'|'DISABLED'}`. Empty array = no maps currently on the vehicle. The
  currently active map for the AGV's reported position is
  `state.agvPosition.mapId`.

## 8. Actions

Actions execute via the `action[]` array on a node or edge, or standalone
via `instantActions`. Edge actions only run while the AGV is physically on
that edge (Section 6.10.2 governs exact trigger timing — see §9 below). Node
actions may run as long as needed but should ideally be self-terminating
(e.g. a 5s audio signal) or come in start/stop pairs (e.g.
`activateWarningLights` / `deactivateWarningLights`). If an AGV can't map a
needed behavior to a predefined action, the manufacturer may define
additional custom action types for master control to use.

### 8.1 Predefined actions

| action | counter action | description | idempotent | scope (instant/node/edge) | key parameters |
|---|---|---|---|---|---|
| `startPause` | `stopPause` | Activates pause mode; vehicle/actions can usually still continue once resumed; order resumable. Linked state: `.paused`. | yes | instant only | – |
| `stopPause` | `startPause` | Deactivates pause mode, resumes movement/actions; can also undo a hardware-button pause if configured. | yes | instant only | – |
| `startCharging` | `stopCharging` | Activates charging (standing or driving on a charging lane); overcharge protection is the vehicle's responsibility. Linked state: `.batteryState.charging`. | yes | instant, node | – |
| `stopCharging` | `startCharging` | Deactivates charging to allow a new order; can also be triggered by the vehicle/station itself (e.g. full battery). | yes | instant, node | – |
| `initPosition` | – | Resets/overrides the AGV's pose. Linked state: `.agvPosition.{x,y,theta,mapId}`, `.lastNodeId`, `.maps`. | yes | instant, node (elevator) | `x`,`y`,`theta` (float64), `mapId` (string), `lastNodeId` (string) |
| `enableMap` | – | Enables a previously downloaded map without repositioning. Linked state: `.maps`. | yes | instant, node | `mapId`, `mapVersion` |
| `downloadMap` | – | Triggers download of a new map; errors reported in state; finishes after verifying download + preparing the map. Linked state: `.maps`. | yes | instant only | `mapId`, `mapVersion`, `mapDownloadLink`, optional `mapHash` |
| `deleteMap` | – | Triggers removal of a map from vehicle memory. Linked state: `.maps`. | yes | instant only | `mapId`, `mapVersion` |
| `stateRequest` | – | Requests the AGV send a fresh state report. | yes | instant only | – |
| `logReport` | – | Requests the AGV generate/store a log report. | yes | instant only | `reason` (string) |
| `pick` | `drop` (if automated) | Request to pick a load. Multi-LHD vehicles must include `lhd`. Linked state: `.load`. | no | node, edge | `lhd?`, `stationType?`, `stationName?`, `loadType`, `loadId?`, `height?`, `depth?`, `side?` (conveyor) |
| `drop` | `pick` (if automated) | Request to drop a load (see `pick` for parameter details). Linked state: `.load`. | no | node, edge | same family as `pick` |
| `detectObject` | – | AGV detects an object (load, charging spot, free parking position). | yes | node, edge | `objectType?` |
| `finePositioning` | – | Position exactly on a target — deviates from the node position if needed (node), or aligns on stationary equipment while traversing (edge). | yes | node, edge | `stationType?`, `stationName?` |
| `waitForTrigger` | – | AGV waits for an external/manual trigger (e.g. button press, manual loading); master control owns timeout/cancel logic. | yes | node only | `triggerType` |
| `cancelOrder` | – | Stops the AGV ASAP (immediately or at next node) and deletes the order; all actions canceled. | yes | instant only | – |
| `factsheetRequest` | – | Requests the AGV send its `factsheet`. | yes | instant only | – |

### 8.2 Action state lifecycle (`actionStatus`)

```
        WAITING
       /   |    \
  (instant starts) (needs prep) (instant succeeds)
      |      ↓          |
      |  INITIALIZING   |
      |   ↓    ↑(pause/unpause)
      |  RUNNING ←→ PAUSED
      |   ↓        ↓
      ↓  FINISHED  FAILED
   (any state can also reach FAILED via cancel/abort,
    or FINISHED/FAILED depending on outcome)
```

Enum values and meaning:

| `actionStatus` | Description |
|---|---|
| `WAITING` | Action received, but its triggering node/edge not yet reached/entered. |
| `INITIALIZING` | Action triggered; preparatory steps in progress (e.g. pre-lift, powering a sensor). |
| `RUNNING` | Action actively executing. |
| `PAUSED` | Paused via a pause instantAction or an external trigger (e.g. physical pause button). |
| `FINISHED` | Done; result (if any) reported via `resultDescription`. |
| `FAILED` | Could not finish, for any reason. |

> An order containing an action the AGV cannot handle at all must be
> **rejected outright** — it must never be reported as `FAILED` once
> accepted.

### 8.3 Blocking types & multi-action handling

| `blockingType` | Behavior |
|---|---|
| `NONE` | Runs in parallel with other actions and with driving. |
| `SOFT` | Runs in parallel with other actions, but the vehicle must not drive. |
| `HARD` | No other action may run in parallel; vehicle must not drive. |

When multiple actions exist on one node: iterate the action list; `NONE`
actions get queued into a "parallel execution list" and run together (no
need to stop driving); a `SOFT` action stops driving, then is added to the
parallel list; a `HARD` action stops driving, first flushes/runs whatever is
queued in the parallel list, then runs alone before continuing to the next
list item. After the list is exhausted, any remaining parallel-list actions
are executed before continuing order handling.

## 9. Topic: `state` (AGV → master control)

The entire AGV state is sent on **one** topic (combining order progress,
battery, errors, etc.) to minimize broker/master-control workload and keep
everything synchronized. Published on relevant events, or at least every
**30s**.

Trigger events: receiving an order; receiving an order update; load status
change; errors/warnings; driving over a node; operating-mode switch; change
in `driving`; change in `nodeStates`/`edgeStates`/actionStates/maps. Related
events (e.g. a new order forcing a nodeStates/edgeStates update) should be
coalesced into a single state publish rather than several.

### 9.1 Concept & logic

`nodeStates`/`edgeStates` track everything still left to traverse; if the
AGV can self-localize, it also reports `agvPosition` (and optionally
`trajectory`, if it self-plans and that's been agreed during integration —
once nodes are released, the AGV may not change its reported trajectory).

### 9.2 Traversal semantics (when is a node/edge "done"?)

The AGV decides for itself when a node counts as traversed — generally, its
control point must be within the node's `allowedDeviationXY`/
`allowedDeviationTheta` (and, if the *next* edge has a `corridor`, within
that too).

- On traversal: remove that node's entry from `nodeStates`; set
  `lastNodeId`/`lastNodeSequenceId` to the traversed node's values; trigger
  that node's actions.
- Traversal of node N also marks **leaving** the edge that led into it: that
  edge is removed from `edgeStates`, and any actions that were active on it
  are finished.
- Traversal of node N also marks **entering** the following edge (if any):
  that edge's actions now trigger — *except* if the AGV must pause on the
  edge (soft/hard blocking action, or otherwise), in which case it only
  "enters" the edge once it resumes moving.

### 9.3 Other state-topic semantics

- **Base request**: AGV sets `newBaseRequest: true` when its base is running
  low, to prompt master control to extend it (avoiding unnecessary slowdown).
- **Information**: AGV may push arbitrary `information[]` entries for
  visualization/debugging — **master control must never use these for
  logic**.
- **Errors**: reported via `errors[]`; `errorLevel` is `WARNING`
  (self-resolving, e.g. a transient field violation) or `FATAL` (needs human
  intervention); `errorReferences[]` cross-link to the relevant
  node/edge/order/action IDs.

### 9.4 `state` message structure

Top-level (after header): `maps[map]` (array — currently stored maps),
`orderId` (string — kept until a new order arrives; `""` if none yet),
`orderUpdateId` (uint32, `0` if none), optional `zoneSetId`, `lastNodeId`
(string, `""` if none), `lastNodeSequenceId` (uint32, `0` if none),
`nodeStates[nodeState]`, `edgeStates[edgeState]`, optional `agvPosition`,
optional `velocity`, optional `loads[load]`, `driving` (bool — true if
driving and/or rotating; excludes e.g. lift movement), optional `paused`
(bool), optional `newBaseRequest` (bool), optional `distanceSinceLastNode`
(m, float64 — for line-guided AGVs), `actionStates[actionState]`,
`batteryState`, `operatingMode` (string enum), `errors[error]`, optional
`information[info]`, `safetyState`.

**map**: `mapId`, `mapVersion` (string), optional `mapDescription`,
`mapStatus` (`ENABLED`|`DISABLED`).

**nodeState**: `nodeId`, `sequenceId`, optional `nodeDescription`,
`released` (bool), optional `nodePosition` (same shape as in the order
topic — optional here too, master control already has it; useful for
debugging).

**edgeState**: `edgeId`, `sequenceId`, optional `edgeDescription`,
`released`, optional `trajectory` (NURBS — segments start where the AGV
*enters* the edge and terminate where it reports the end node traversed).

**agvPosition**: `positionInitialized` (bool), optional `localizationScore`
(float64 [0..1] — logging/visualization only), optional `deviationRange` (m
— logging/visualization only), `x`, `y` (m, float64), `theta` (float64
[-Pi..Pi]), `mapId` (string), optional `mapDescription`.

**velocity**: optional `vx`, `vy` (m/s), optional `omega` (rad/s) — all in
vehicle coordinates.

**load**: optional `loadId` (string — empty if unidentified), optional
`loadType`, optional `loadPosition` (e.g. "front"/"back"/"positionC1"),
optional `boundingBoxReference`, optional `loadDimensions`, optional
`weight` (kg, float64 ≥0).

**boundingBoxReference**: `x`, `y` (float64), `z` (float64), optional
`theta` (float64) — reference point is the center of the bounding box's
bottom face, in AGV coordinates.

**loadDimensions**: `length`, `width` (m, float64), optional `height` (m).

**actionState**: `actionId` (string), optional `actionType` (informational
only — master control already knows it from the order), optional
`actionDescription`, `actionStatus` (enum, §8.2), optional
`resultDescription` (string).

**batteryState**: `batteryCharge` (%, float64 — vehicles with only
good/bad granularity report 80%/20%), optional `batteryVoltage` (V),
optional `batteryHealth` (%, int8 [0..100]), `charging` (bool), optional
`reach` (m, uint32 — estimated remaining range at current charge).

**error**: `errorType` (string), `errorReferences[errorReference]`,
optional `errorDescription`, optional `errorHint`, `errorLevel`
(`WARNING`|`FATAL`).

**errorReference** / **infoReference**: `{referenceKey: string,
referenceValue: string}` (e.g. `{"nodeId": "node7"}`).

**info**: `infoType` (string), `infoReferences[infoReference]`, optional
`infoDescription`, `infoLevel` (`DEBUG`|`INFO`).

**safetyState**: `eStop` (`AUTOACK`|`MANUAL`|`REMOTE`|`NONE`),
`fieldViolation` (bool — protective field currently violated).

### 9.5 `operatingMode` values

| Value | Meaning |
|---|---|
| `AUTOMATIC` | Full master-control control; AGV drives/acts on orders. |
| `SEMIAUTOMATIC` | Master control issues orders/actions, but a human-operated HMI bounds the driving speed (can't exceed automatic-mode speed); steering stays automatic (non-safe HMI possible). |
| `MANUAL` | Master control has no control; HMI drives/steers/operates load handling directly; AGV still reports location; entering/leaving this mode clears all orders (requires a safe HMI). |
| `SERVICE` | Master control has no control; authorized personnel can reconfigure the AGV. |
| `TEACHIN` | Master control has no control; the AGV is being taught (e.g. mapping) by an operator/master control tool. |

Near-real-time position/velocity broadcast for visualization purposes only —
same `agvPosition`/`velocity` object shapes as in `state`. Update rate is up

## 11. Topic: `connection` (broker/AGV → master control)

Liveness/heartbeat signal using **MQTT last-will**, QoS **1**. Suggested
topic: `uagv/v2/manufacturer/SN/connection`. All messages on this topic must
be sent with the **retained** flag.

`connectionState` enum:
- `ONLINE` — active connection.
- `OFFLINE` — gracefully ended (AGV explicitly published `OFFLINE` then
  issued a clean MQTT disconnect).
- `CONNECTIONBROKEN` — the broker's last-will fired because the connection
  dropped unexpectedly (no graceful disconnect).

Graceful shutdown sequence: AGV publishes `connectionState: OFFLINE` →
disconnects cleanly. Coming back online: AGV (re)sets its last-will to
`CONNECTIONBROKEN` at connect time → publishes `connectionState: ONLINE`.
Heartbeat/keepalive interval is broker-configurable, suggested ~15s. Note:
because of how MQTT last-will works, the last-will message's own
`timestamp`/`headerId` are necessarily stale by the time it's actually sent.

## 12. Topic: `factsheet` (AGV → master control)

A largely static, human- *and* machine-readable description of an AGV type
series — used for planning, dimensioning, simulation, and integration. Master
control can request a fresh one via the `factsheetRequest` instantAction.
Must be sent with the **retained** flag.

Top-level fields (after header): `typeSpecification`, `physicalParameters`,
`protocolLimits`, `protocolFeatures`, `agvGeometry`, `loadSpecification`,
optional `vehicleConfig`.

**typeSpecification**: `seriesName`, optional `seriesDescription`,
`agvKinematic` (`DIFF`|`OMNI`|`THREEWHEEL`), `agvClass`
(`FORKLIFT`|`CONVEYOR`|`TUGGER`|`CARRIER`), `maxLoadMass` (kg),
`localizationTypes[]` (e.g. `NATURAL`, `REFLECTOR`, `RFID`, `DMC`, `SPOT`,
`GRID`), `navigationTypes[]` priority-ordered (e.g.
`PHYSICAL_LINE_GUIDED`, `VIRTUAL_LINE_GUIDED`, `AUTONOMOUS`).

**physicalParameters**: `speedMin`/`speedMax` (m/s), optional
`angularSpeedMin`/`angularSpeedMax` (rad/s), `accelerationMax`/
`decelerationMax` (m/s², at max load), `heightMin`/`heightMax` (m), `width`,
`length` (m).

**protocolLimits**: `maxStringLens{ msgLen, topicSerialLen, topicElemLen,
idLen, idNumericalOnly, enumLen, loadIdLen }` (all uint32 except the bool),
`maxArrayLens{ order.nodes, order.edges, node.actions, edge.actions,
actions.actionsParameters, instantActions, trajectory.knotVector,
trajectory.controlPoints, state.nodeStates, state.edgeStates, state.loads,
state.actionStates, state.errors, state.information,
error.errorReferences, information.infoReferences }` (all uint32),
`timing{ minOrderInterval, minStateInterval, defaultStateInterval?,
visualizationInterval? }` (all float32, seconds). A limit of `0`/undefined
means "no explicit limit".

**protocolFeatures**: `optionalParameters[optionalParameter]` — each
`{parameter: string (e.g. "order.nodes.nodePosition.allowedDeviationTheta"),
support: 'SUPPORTED'|'REQUIRED', description?}`; `agvActions[agvAction]` —
every action (predefined + manufacturer-specific) the AGV supports: each
`{actionType, actionDescription?, actionScopes[]:
'INSTANT'|'NODE'|'EDGE', actionParameters[{key, valueDataType:
'BOOL'|'NUMBER'|'INTEGER'|'FLOAT'|'STRING'|'OBJECT'|'ARRAY', description?,
isOptional?}], resultDescription?, blockingTypes[]:
'NONE'|'SOFT'|'HARD'}`.

**agvGeometry**: `wheelDefinitions[wheelDefinition]` — each `{type:
'DRIVE'|'CASTER'|'FIXED'|'MECANUM', isActiveDriven, isActiveSteered,
position:{x,y,theta?}, diameter, width, centerDisplacement?, constraints?}`;
`envelopes2d[envelope2d]` — each `{set, polygonPoints[{x,y}], description?}`
(closed, non-self-intersecting polygon); `envelopes3d[envelope3d]` — each
`{set, format (e.g. DXF), data?, url?, description?}`.

**loadSpecification**: `loadPositions[]` (valid values for
`state.loads[].loadPosition` and the `lhd` action parameter — empty/missing
= AGV has no load handling device), `loadSets[loadSet]` — each `{setName,
loadType, loadPositions[]?, boundingBoxReference, loadDimensions,
maxWeight, minLoadhandlingHeight?, maxLoadhandlingHeight?,
minLoadhandlingDepth?, maxLoadhandlingDepth?, minLoadhandlingTilt?,
maxLoadhandlingTilt?, agvSpeedLimit?, agvAccelerationLimit?,
agvDecelerationLimit?, pickTime?, dropTime?, description?}`.

**vehicleConfig**: `versions[versionInfo]` — each `{key, value}`
(software/hardware version pairs, e.g. `softwareVersion: v1.12.4-beta`);
optional `network{ dnsServers[], ntpServers[], localIpAddress?, netmask?,
defaultGateway? }` — network info should not change while operating.

## 13. Best practice (Section 7 of the spec)

### 13.1 Error reference

When an order causes an error, the AGV should populate `errorReferences[]`
with whatever is relevant: `headerId`; the topic that caused it (`order` or
`instantAction`); `orderId`/`orderUpdateId` if an order update caused it;
`actionId` if an action caused it; the list of offending parameters if bad
action parameters caused it. If an action fails due to external factors
(e.g. expected load missing), reference its `actionId`.

### 13.2 Parameter format

`actionParameter`/`errorReference`/`infoReference`-style values are encoded
as an array of `{key, value}` objects rather than arbitrary nested JSON, to
keep the schema generic — `value` may be any JSON type (array, boolean,
number, string, object). Example:

```json
"actionParameters": [
  {"key": "stationType", "value": "floor"},
  {"key": "weight", "value": 8.5},
  {"key": "loadType", "value": "pallet_eu"}
]
```
